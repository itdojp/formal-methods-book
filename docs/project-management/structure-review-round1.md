# 1. 書籍の概要

- 想定読者（`docs/index.md` の記述ベース）
  - 形式的手法（仕様記述・模型検査・定理証明）の全体像を掴み、実務での部分導入まで含めて検討したいエンジニア。
  - 既にソフトウェア開発の基本（テスト・レビュー等）は理解しており、「なぜ形式化が必要か」「どこに適用すべきか」を判断したい読者。
- 本書で達成しようとしているゴール（`docs/index.md` の記述ベース）
  - Alloy / Z / CSP / TLA+ 等の位置づけを整理し、適用判断の観点を持つ。
  - 小規模な仕様を対象に、仕様記述→模型検査/性質検証までの流れを体験する。
  - 「全面導入」ではなく、設計レビュー・テスト設計・ドキュメント改善などへの部分適用も含めた導入シナリオを描けるようにする。

## 3. 構成上の気になる点（候補）

- 学習目標の明示が弱い
  - 各章の冒頭に「ミニ要約」はあるが、Stage2要件（章の学習目標と節内容の対応）としてはチェック可能な形になっていない。
- 「導入 → 本論 → まとめ」の視認性
  - 章末は課題・演習で閉じる構成が多く、「まとめ」相当の整理が見出しとして明示されていない章がある。
- 節番号の重複（第12章）
  - 第12章内で `## 12.3` と `## 12.4` が重複して登場する。章内参照や読者の位置把握を阻害する可能性が高い。
- 本文ソースの二重管理の可能性（運用面）
  - `docs/chapters/**` と `docs/src/**` が共存しており、後者に内容が異なるページが含まれている。どれを正本として維持するかが不明確だと更新漏れのリスクになる（要確認）。

## 4. 構成改善の提案（案）

- 章頭の学習目標を「チェック可能」な形で追加する
  - 各章冒頭に 3〜5 点の「この章でできるようになること」を置き、本文の節と対応づける（例: 章末課題の観点を逆算して目標化する）。
- 章末に「まとめ」節を追加し、課題の前に配置する
  - 章末の課題・演習の前に `## まとめ` を置き、重要概念・判断軸・次章への接続を箇条書きで固定化する。
- 第12章の節番号重複を解消する
  - 既存の節（CI/運用、検証ツール、コード生成）を整理し、12.3/12.4 を再採番する（章内参照の信頼性向上）。
- `docs/src/**` の位置づけを明確化する（運用面）
  - `docs/src/**` を参照しない方針なら、Jekyll の `exclude` への追加やディレクトリ移動（例: `archive/`）を検討する（要確認）。
